# QA 记录

## Q1: 如何减少在BOSS直聘上筛选简历的时间成本？

**日期**: 2026-03-14

**问题**: 每天在BOSS直聘投入大量时间找合适的简历，有什么解决方案？

**回答**:

### 策略层面
- 重写JD，加入"劝退条件"减少不匹配的候选人
- 提高薪资竞争力，减少无效沟通
- 使用平台付费功能（牛人推荐等）

### 技术层面（已实施）
开发了油猴脚本 `boss-helper.user.js`，实现：
- **XHR拦截**：从API响应中获取候选人结构化数据
- **自动评分**：基于学历、院校(985/211)、工作年限、技能关键词等维度打分
- **颜色高亮**：绿色(高匹配)、黄色(一般)、灰色半透明(不匹配)
- **配置面板**：可视化编辑规则、关键词、阈值，支持导入导出

### 使用方式
1. 安装 Tampermonkey 浏览器扩展
2. 导入 `boss-helper.user.js` 脚本
3. 正常登录BOSS直聘浏览候选人，脚本自动增强页面

## Q2: 油猴脚本实现中发现了哪些Bug并如何修复？

**日期**: 2026-03-14

**问题**: 基于Plan实现的boss-helper.user.js有哪些关键缺陷？

**回答**:

### 已修复的5个问题

1. **关键词独立计分（核心Bug）** — 计划要求"每个匹配关键词+10分"，但代码用`containsAny`只返回true/false，匹配3个关键词也只加10分。修复：在`scoreCandidate`中计算`matchedKeywords.length`并乘以单次分值。

2. **标签渲染innerHTML注入** — `createKeywordSection`用`innerHTML`直接拼接keyword文本，含HTML字符的关键词会破坏DOM。修复：改用`textContent` + `appendChild`。

3. **规则编辑器硬编码索引** — `showRuleEditor`用`querySelectorAll[2]`定位section，顺序变化会出错。修复：用`data-bh-section="rules"`属性精确定位。

4. **renderCard死代码** — 开头检查`data-bhScored`就return，后面的"清除旧渲染"代码永不执行。修复：移除死代码，tooltip始终显示并展示关键词匹配详情。

5. **配置面板滚动穿透** — 面板打开时背景页面仍可滚动。修复：打开时`body.style.overflow='hidden'`。

## Q3: 能否对高分候选人自动打招呼？

**日期**: 2026-03-14

**问题**: 对超过配置分数的候选人，能否自动打招呼或通知我？

**回答**:

### 自动打招呼 — 不做
- **封号风险**：BOSS直聘有反自动化检测，模拟点击打招呼是典型机器人行为
- **权益浪费**：每天仅20次打招呼机会，评分系统基于有限信息的粗筛，误判率不低
- **用户体验差**：模板消息让候选人一看就知道是群发

### 已实现的替代方案 — 通知系统
- **页面内Toast通知**：右上角弹出候选人摘要（姓名、分数、学历、学校），含"定位"和"忽略"按钮
- **桌面通知**：通过`GM_notification`推送系统通知，点击可跳转到候选人卡片
- **提示音**：短促的880Hz正弦波提示
- **呼吸动画**：高分卡片出现时有3秒脉冲边框动画吸引注意
- **通知分数下限**：可在右下角配置面板「高级设置」中设置，仅当候选人分数达到该阈值时才通知，且不会低于绿色高匹配阈值
- 所有通知功能可在配置面板「高级设置」中开关

## Q4: 为什么部分候选人的长空窗期没有被扣分？

**日期**: 2026-03-18

**问题**: 页面上明明能看到工作时间段，但脚本没有对明显的 gap 时间过长进行扣分。

**回答**:

### 根因
- **旧配置/远程角色配置缺规则**：`work_gap` 是后续新增规则，早期保存的本地配置和部分远程 `configs/*.json` 里没有这条规则，导致即使算出了 `gapCount/maxGapMonths` 也不会扣分
- **时间格式识别过窄**：原来的兜底正则只识别 `YYYY.MM - YYYY.MM`，对 `YYYY年MM月`、`至今`、`Present` 等格式不够稳健

### 修复
- 在配置合并和远程角色同步时，自动回填缺失的 `work_gap` 规则
- 扩展工作时间解析，支持中文年月与“至今/现在/Present/Current”等结束时间
- 同步更新各角色远程配置文件，避免新机器首次加载时继续漏扣

## Q5: 高分候选人通知为什么会挡住主界面的分数徽章？

**日期**: 2026-03-20

**问题**: 页面右上角的高分候选人 Toast 停留时间偏长、内容过宽，容易遮挡卡片上的分数徽章。

**回答**:

### 根因
- Toast 采用横向布局，正文和按钮同时占宽，学校/薪资等信息一长就会把弹窗整体拉宽
- 自动消失时间固定为 8 秒，在连续出现高分候选人时遮挡时间过久

### 修复
- 将 Toast 宽度进一步压缩到更窄的竖向卡片，减少对右侧主界面的横向遮挡
- 保留学校、经验、薪资等摘要信息，改为多行紧凑显示
- 收紧内边距、宽度和按钮尺寸，限制最大宽度以降低遮挡面积
- 自动消失时间从 8 秒缩短到 4.5 秒

## Q6: 为什么“搜索牛人”里的“最新”子 tab 没有显示评分？

**日期**: 2026-03-20

**问题**: 同一页面下“推荐”子 tab 可以正常显示分数，但切到“最新”后卡片没有被评分和高亮。

**回答**:

### 根因
- 非激活子 tab 的卡片可能已提前渲染在 DOM 中，但初始处于隐藏状态；旧逻辑会把这类卡片标记为 `skip`
- 用户切换子 tab 时，旧逻辑没有补一次重处理，隐藏卡片即使变为可见也不会重新评分
- 接口识别过于宽松，`/wapi/zpblock/recommend/filters` 这类筛选接口也可能被误判成候选人数据
- 卡片选择器优先返回“数量最多”的列表，切换到“最新”后仍可能继续抓到隐藏的“推荐”列表节点

### 修复
- 对暂时隐藏的卡片改为标记 `pending`，不再永久跳过，变为可见后可重新处理
- 新增 tab/按钮点击后的延迟重处理，覆盖“推荐/最新”这类子 tab 切换场景
- 将卡片最小宽度阈值从 `200` 下调到 `120`，兼容更窄的列表布局
- 候选人接口提取改为递归评分选择，优先识别真正带身份与履历字段的候选人对象，降低筛选接口误命中
- 卡片检测改为优先返回当前可见列表，不再拿隐藏 tab 的卡片做评分
